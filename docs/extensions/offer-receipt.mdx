The Offer & Receipt extension adds server-side signatures to x402, enabling cryptographic proof of payment terms and service delivery.

## Overview

This extension provides two signed artifacts:

- **Signed Offers** - The resource server cryptographically commits to payment terms presented in `accepts[]`
- **Signed Receipts** - After successful payment and service delivery, the server returns a signed receipt

These artifacts enable:
- Dispute evidence and auditability
- User review attestations ("Verified Purchase" badges)
- Verifiable proof of commercial interactions for reputation systems
- Agent-to-agent commerce with machine-verifiable proof

## Status

This extension is **optional** and works with both x402 v1 and v2. The behavioral requirements are stable, but wire format may evolve as the x402 extension architecture standardizes.

## Signature Formats

Two formats are supported:

- **EIP-712** - Best for wallet-based signing (MetaMask, WalletConnect, etc.)
- **JWS** - Best for server-side signing with managed keys (HSM, KMS, etc.)

## Server Usage

### Basic Setup

<Tabs>
  <Tab title="TypeScript">
    ```typescript
    import { x402ResourceServer } from "@x402/core/server";
    import { 
      createOfferReceiptExtension,
      createJWSOfferReceiptIssuer,
    } from "@x402/extensions/offer-receipt";

    // Create an issuer (JWS or EIP-712)
    const issuer = createJWSOfferReceiptIssuer(
      "did:web:api.example.com#key-1", 
      jwsSigner
    );

    // Register the extension
    const server = new x402ResourceServer(facilitator)
      .registerExtension(createOfferReceiptExtension(issuer));

    // Declare in route config
    const routes = {
      "GET /api/data": {
        accepts: { 
          payTo, 
          scheme: "exact", 
          price: "$0.01", 
          network: "eip155:8453" 
        },
        extensions: {
          "offer-receipt": {
            info: {
              includeTxHash: false // Privacy-minimal receipts
            }
          }
        }
      }
    };
    ```
  </Tab>
</Tabs>

### Creating Signers

<Tabs>
  <Tab title="JWS Signer">
    ```typescript
    import { createJWSOfferReceiptIssuer } from "@x402/extensions/offer-receipt";
    import { SignJWT } from "jose";

    // Your JWS signing function
    const jwsSigner = async (payload: object) => {
      const jwt = await new SignJWT(payload)
        .setProtectedHeader({ 
          alg: "ES256K", 
          kid: "did:web:api.example.com#key-1" 
        })
        .sign(privateKey);
      return jwt;
    };

    const issuer = createJWSOfferReceiptIssuer(
      "did:web:api.example.com#key-1",
      jwsSigner
    );
    ```

## Client Usage

### Using wrapFetchWithPayment

The `wrapFetchWithPayment` wrapper can capture offers and receipts through hooks:

```typescript
import { wrapFetchWithPayment, x402Client, x402HTTPClient } from "@x402/fetch";
import {
  extractOffersFromPaymentRequired,
  decodeSignedOffers,
  extractReceiptFromResponse,
  type DecodedOffer,
} from "@x402/extensions/offer-receipt";

const client = new x402Client();
const httpClient = new x402HTTPClient(client);

// Store offers for later matching with receipt
let capturedOffers: DecodedOffer[] = [];

// Capture offers in onPaymentRequired hook
httpClient.onPaymentRequired(async ({ paymentRequired }) => {
  const offers = extractOffersFromPaymentRequired(paymentRequired);
  capturedOffers = decodeSignedOffers(offers);
});

// Create payment-enabled fetch
const fetchWithPay = wrapFetchWithPayment(fetch, httpClient);

// Make request (payment handled automatically)
const response = await fetchWithPay(url);

// Extract receipt from response headers
const receipt = extractReceiptFromResponse(response);
```

### Raw Flow

For full control over offer selection, use the raw flow. See the [Offer/Receipt Example](https://github.com/coinbase/x402/tree/main/examples/typescript/clients/offer-receipt) for a complete working implementation.

The example demonstrates:
1. Making a request and receiving a 402 with signed offers
2. Extracting and decoding offers to inspect payment options
3. Selecting an offer and finding the matching `accepts[]` entry
4. Making payment and receiving a signed receipt
5. Verifying the receipt payload

## Payload Structure

### Offer Payload

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | number | Yes | Offer payload schema version (currently `1`) |
| `resourceUrl` | string | Yes | The paid resource URL |
| `scheme` | string | Yes | Payment scheme identifier (e.g., "exact") |
| `network` | string | Yes | Blockchain network (CAIP-2 format, e.g., "eip155:8453") |
| `asset` | string | Yes | Token contract address or "native" |
| `payTo` | string | Yes | Recipient wallet address |
| `amount` | string | Yes | Required payment amount |
| `validUntil` | number | Optional | Unix timestamp (seconds) when the offer expires |

### Receipt Payload

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | number | Yes | Receipt payload schema version (currently `1`) |
| `network` | string | Yes | Blockchain network (CAIP-2 format) |
| `resourceUrl` | string | Yes | The paid resource URL |
| `payer` | string | Yes | Payer identifier (wallet address) |
| `issuedAt` | number | Yes | Unix timestamp (seconds) when receipt was issued |
| `transaction` | string | Optional | Blockchain transaction hash |

The receipt is **privacy-minimal** by default and omits transaction references to reduce correlation risk. Include `transaction` when stronger verifiability is preferred over privacy.

## Using Receipts as Proofs

Signed receipts serve as cryptographic proofs of commercial transactions. These proofs can be submitted to downstream trust and reputation platforms:

- **[OMATrust](https://github.com/oma3dao/omatrust-docs)** - Decentralized reputation system for verified user reviews and service attestations
- **[PEAC Protocol](https://github.com/peacprotocol/peac)** - Payment Evidence and Attestation Chain for commercial transaction proofs

Integration libraries for these platforms will be added in future releases.

## Security Considerations

The `extractPayload()` functions extract payloads without verifying the signature or checking signer authorization. This is by design — signer authorization requires resolving key bindings (did:web documents, attestations, etc.) which varies by deployment.

For production use, downstream trust systems verify:
1. The signature is valid (EIP-712 or JWS)
2. The signing key is authorized for the resource domain

### Key-to-Domain Binding

To establish trust, bind the signing key's DID to the resource domain using:

1. **`did:web` DID Document** - Serve at `https://example.com/.well-known/did.json`
2. **DNS TXT Record** - Add a TXT record binding a DID to the domain
3. **Key Binding Attestation** - Create an attestation specifying the key's purpose and authorized domain

### Key Management

For production deployments:

- **JWS signing**: Use HSM or KMS-backed keys. The `kid` in the JWS header should be a DID URL that resolves to the public key.
- **EIP-712 signing**: The signing wallet should be the `payTo` address, or have an on-chain/off-chain authorization linking it to the service.
- **Key rotation**: Update DID documents or attestations when rotating keys. Old receipts remain valid if the key was authorized at issuance time.

## Privacy Considerations

- Receipts are minimal by default — they omit transaction references to reduce correlation risk
- Servers may include the optional `transaction` field when verifiability is more important than privacy
- Offers reveal economic terms (amount, asset, payTo address)
- Consider privacy implications when deciding which artifacts to include in public attestations
